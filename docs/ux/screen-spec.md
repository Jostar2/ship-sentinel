# Screen Spec

## 1. Landing

- purpose: explain the service offer in one screen
- blocks
  - hero
  - why QA audit matters now
  - deliverables overview
  - package summary
  - sample release memo
  - CTA

## 2. Client Intake

- purpose: define the audit scope before testing starts
- blocks
  - service identity
  - environment and access
  - critical flows
  - risk areas
  - browser and device targets
  - save and start actions

## 3. Dashboard

- purpose: show what the delivered audit pack looks like
- blocks
  - run summary
  - ship / fix / block decision
  - scenario coverage
  - blocker cards
  - warning cards
  - next owner summary

## 4. QA Session

- purpose: keep almost all operator work in one dominant screen
- blocks
  - persistent context bar
  - screen rail
  - scenario rail
  - reference/live compare
  - capture inbox
  - issue composer
  - spec drift lane
  - evidence timeline
  - package readiness strip

## 5. Deliverable Packager

- purpose: generate and package final client outputs
- blocks
  - export checklist
  - spreadsheet generator
  - memo export
  - package history

## 6. Maintenance Board

- purpose: handle overflow current-state updates and revised artifact management
- blocks
  - request queue
  - revised source upload
  - status tracking
