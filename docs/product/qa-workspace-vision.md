# QA Workspace Vision

## Product intent

이 제품은 단순한 버그 등록 툴이 아니다.

핵심 목적은 아래 흐름을 한 워크스페이스 안으로 묶는 것이다.

- 화면설계서
- 확정 디자인
- 실제 서비스 화면
- 테스트 실행
- 증적 수집
- 이슈 등록
- 현행화 요청
- 수정본 재정리와 재배포

즉, `QA 작업 OS`를 만든다.

## Why this matters

현재 QA 업무가 귀찮고 번거로운 이유는 테스트 자체보다 연결 작업이 많기 때문이다.

- 설계서는 따로 열어야 한다
- 피그마는 따로 확인해야 한다
- 실제 서비스는 브라우저에서 따로 봐야 한다
- 스크린샷은 찍고 저장하고 붙여넣어야 한다
- 이슈 등록 시 같은 정보를 반복 입력한다
- 문서 현행화 요청은 별도 채널로 던져야 한다
- 수정본을 다시 모으고 묶는 작업도 수작업이다

이 제품은 이런 불필요한 연결 비용을 제거해야 한다.

## Core product definition

이 제품은 다음 네 모듈로 생각할 수 있다.

### 1. Reference Sync

- 화면 ID 기준으로 설계서, 피그마, 확정 디자인, 실제 화면 연결
- 한 화면에서 레퍼런스와 실제 상태 비교
- 버전과 변경 이력 추적

### 2. Execution Workspace

- 테스트 시나리오 실행
- 현재 문맥을 유지한 채 증적 수집
- 스크린샷과 화면 정보 자동 귀속
- 실시간 이슈 생성

### 3. Issue + Change Request

- 증적 기반 이슈 등록
- 화면 ID, 환경, URL, 브라우저 정보 자동 입력
- 현행화 요청을 이슈 흐름에서 바로 생성

### 4. Document Maintenance

- 현행화 요청 목록 관리
- 수정본 업로드 또는 편집
- 변경 이력 추적
- 최종 문서/산출물 재패키징

## Product principles

### Zero Re-upload

- 스크린샷과 증적을 다시 업로드하지 않게 한다

### Zero Re-context

- 동일한 메타데이터를 반복 입력하지 않게 한다

### Evidence First

- 모든 이슈는 증적과 연결되어야 한다

### One Screen Rule

- QA 작업 중 창 전환을 최소화한다

### Doc Sync Native

- 현행화 요청은 부가 기능이 아니라 핵심 플로우다

## UX thesis

가장 중요한 화면은 `Screen Workspace`다.

이 화면 안에 최소한 아래가 같이 있어야 한다.

- 왼쪽: 설계서 / 피그마 / 확정 디자인 참조
- 가운데: 실제 서비스 화면 또는 테스트 대상
- 오른쪽: 이슈 등록 / 현행화 요청 패널
- 하단: 증적 타임라인 / 변경 이력 / 코멘트

이슈 생성은 10초 안에 끝나야 한다.

- 스크린샷 자동 첨부
- 화면 ID 자동 선택
- URL / 브라우저 / 해상도 자동 입력
- severity와 제목, 설명만 적으면 등록
- 필요 시 현행화 요청 생성 체크

## Platform thesis

이 제품은 순수 웹앱만으로는 부족하다.

원하는 UX를 만들려면 `웹 앱 + 윈도우 에이전트` 조합이 필요하다.

### Web app

- QA 워크스페이스
- 이슈/증적/현행화 관리
- 문서 버전 관리

### Windows helper

- 클립보드 감지
- 스크린샷 파일 감지
- 활성 창 정보 수집
- 현재 QA 세션과 자동 연결

### Document parser

- PPTX/PDF/Figma 메타 추출
- 화면 ID 인덱싱

### Automation engine

- 향후 회귀 자동화
- UI 비교
- 재검수 지원

## MVP boundary

### Include

- 웹서비스 QA 워크스페이스
- 화면 ID 기반 레퍼런스 연결
- 스크린샷 자동 업로드용 윈도우 헬퍼
- 이슈 등록 + 증적 자동 첨부
- 현행화 요청 생성
- 수정본 업로드 및 버전 관리
- 최종 산출물 재묶기

### Exclude

- 완전 자동 QA
- 모든 포맷 완벽 지원
- 실시간 공동편집
- 광범위한 외부 툴 연동
- 대규모 자동화 인프라

## Recommended initial focus

첫 단계는 아래 조합으로 제한하는 게 적절하다.

- target: 웹서비스 QA
- reference source: Figma + PDF 우선
- issue management: 내장형 우선

PPTX는 초기에 `참조 / 추출 / 재패키징` 수준으로 두고, 직접 편집은 후순위로 둔다.
